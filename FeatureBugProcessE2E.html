The gathering of raw feedback and issues from various external touchpoints, validation, and formalization into the project management system.

  • Intake Sources: Members of the Customer Success (CS), Product teams, and general staff collect feedback via client site visits, Intercom messages, direct calls, and customer surveys.
  • Categorization: The staff member collecting the feedback identifies items broadly as either Bugs (broken functionality) or Feature Requests (new/improved functionality).
  • Replication: The Initial Reporter must attempt to replicate the behavior in both the live Production environment and the Staging environment.
  • Analysis: The Initial Reporter uses tools (e.g., ChatGPT, Browser Inspector) to investigate why the issue is occurring.
  • User Confirmation: Only after steps above are complete should validity be confirmed to the user.
  • Ticket Creation: Reporter fills out Linear ticket with Priority/Customer fields.
  • Urgency Branching:
    • Urgent: Post to #et-general Slack & tag On-Call.
    • Standard: Create in Linear (no Slack).

Deciding when and by whom the work will be done.

  • Urgent Bugs: The On-Call Engineer picks up the ticket immediately.
  • Non-Urgent / Features:
    • Continuous Triage: PMs constantly review the intake queue.
    • Fortnightly Sync: CS and PMs meet to review high-priority items.
  • Tickets moved to:
    • "Needs Improving" (Reporter must re-write)
    • "Backlog" (Work will be queued for fixathon at end of quarter)
    • "To Do" (Work will be picked up imminently)

The execution of the fix or feature build, with different paths for urgent and standard items.

  • Urgent Path: Engineers push code directly to Production immediately, bypassing the standard release cycle. Linear status must be updated as work progresses.
  • Standard Path: Engineers push code to Staging following the standard release process. Linear status must be updated as work progresses (In Progress, PR Review, Merged).

Verifying the fix works before telling the customer.
