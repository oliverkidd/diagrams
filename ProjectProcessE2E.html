The goal of this stage is to move from a broad problem to a specific set of bullet points that define the project boundaries.

  • Brainstorming: PM leads the collection of information regarding the problem, collaborating with the Client Team and users.
  • Scope Definition: PM converts the brainstormed information into a structured list of bullet points outlining the scope of the new project.
  • Scope Review Meeting:
    • PM proposes the scope bullet points to Engineering, CT, and C-Suite.
    • Engineering provides rough estimates for each scope point.
    • Decisions are made on exactly what will be tackled in the project (aligned with quarterly planning).

This stage runs asynchronously initially, culminating in a synchronous agreement on the plan.

  • Drafting & Design: PM creates the draft PRD. UX Team creates wireframes/designs to visualize the solution outlined in the PRD.
  • PRD Review: Relevant parties from the Scope Review (Stage 1) review the document and leave comments to align on the solution.
  • Kick-off Meeting: PM schedules a meeting with Engineering, C-Suite, and UX to walk through the PRD and solution to ensure full alignment.
  • Scheduling: The PRD is split into successive stages. Engineers sign off on estimated delivery dates. Milestones are added to Linear aligned to these dates.

This stage focuses on code quality and preparing the feature for the testing environment.

  • PR Reviews: Lead Engineers review Pull Requests (PRs) and provide coding comments.
  • Merge to Main: Engineers address comments and finalize solutions.
  • Staging Deployment: Engineer notifies PM when PRs are all in Staging and merged to the Main branch.
  • Staging Review: PM reviews the entire project on Staging (required as features are behind Feature Flags).
  • Release Forecasting: On Mondays, Engineers must provide an update on which features they expect to be fully released that week. This triggers the Quality/PM teams to write Edge Cases for the upcoming testing phase.

A staggered release process using Feature Flags to minimize risk.

  • Beta (Early User Feedback): Feature Flag is turned on for a small number of specific clinics/users. Meetings are set up to collect qualitative feedback. If tweaks are required, Eng implements changes.
  • A/B Testing (Scale): Feature Flag is turned on for a larger, random subset (e.g., 50%) of users to test scalability.
  • Full Release: Once PM is satisfied with stability and feedback, the Feature Flag is removed (turned on for everyone).
  • Communication: Product Comms (emails) are sent to the user base explaining the feature. (Note: Early beta testers should receive communication prior to the general blast).
